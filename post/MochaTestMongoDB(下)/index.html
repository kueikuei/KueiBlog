server.test.js

延續MochaTestMongoDB(上)文章

這次我們將鎖定講解server.test.js
瞭解如何透過單元測試對於完整的前後端過程分別進行測試。

測試程式碼在此

測試前準備

require package

supertest專門用來測試express
任何特別的應用程式理論上都會有大神門開發相對應模組進行測試(只要不是太冷門的應用)
這在MochaTestExpress時聊過

註解部分為原作者引用方式
是透過ES6去進行
但筆者這邊會引用失敗故註解

在server.js中我們透過express
將眾多(req,res)與對應的url設定完成
並且module.exports.app = app;出去
等號左邊的app為物件函式名稱；右邊為我們設定好的express app
這邊我們就必須特別使用.app呼叫
不然會呼叫到整個server.js
而app的引入成功至關重要
因爲所有的url與對應method都已經在app中設定好

此外會用到mongodb的ObjectID方法
以及Todo model
一併引用

測試資料

這邊有兩組被陣列todos包覆的資料
在每一個測試案例(專指it)前
我們都先清空Todo.remove({})DB
並且將將這兩筆資料存入MongoDB中以便測試

Post測試

Post
我們進行兩個測試

  • should create a new todo
  • should not create todo with invalid body data

我們用一個測試套件describe包覆這兩個測試案例it

(1) request(app)
request給予app參數
如此一來所有的CRUD method都能使用

(2) .post(‘/todos’)
使用post方法
網址列讀取/todos

(3) .send({text})
這邊我們無法依靠postman傳送req
但透過supertest的send({})
能做到一模一樣的事情
{text}會將{text:'Test todo text'}傳送(send)出去

接著轉入server.js的app.post

接著做的事情相信大家都很熟悉
創建新資料並儲存及回傳(res.send(doc))
MochaTestMongoDB(上)

(4) expect(200)
一旦回傳成功
http state顯示為200
並且加以去驗證expect(200)

(5) expect((res) => {expect(res.body.text).toBe(text);})
接著我們測試存入DB後回傳的res
是否為我們想存入的內容(等於是幫DB先行測試)
最後加入end((err, res)結束

(6) test for DB
這邊我們測試存入DB中的資料
分別藉由測試其長度length以及內容是否正確

以上步驟繁瑣
但是一步又一步完整的測試了post
接著介紹Get的測試

Get測試

Get
分成兩個測試套件
共四個測試案例
我們在這邊分成兩種路徑進行測試
分別是/todos與/todos/:id

GET /todos

先看單純的GET /todos範例

(1).get(‘/todos’)
取得/todos url
方法用get
並轉到server.js的app.get('/todos') method

做的事情很單純
到DB中找尋所有資料
並且res回傳所有資料

(2).expect(200)
取得資料則得到status 200

(3).expect((res) => {expect(res.body.todos.length).toBe(2);})
這邊不比較內容
因為整個資料要比較資料內容會較於繁瑣
故比較資料數量是否正確
這邊輸入兩筆資料
故要等於2

GET /todos/:id範例
進行三個測試案例

  • should return todo doc
  • should return 404 if todo not found
  • should return 404 for non-object ids

依序講解

should return todo doc
是正確取得資料時該做的測試
(1) .get(/todos/${todos[0]._id.toHexString()})
透過${}允許我們在裡頭擺放變數

(2) .expect(200)
轉換到server.js的app.get('/todos/:id') method

server.js的app.get詳細介紹能看
MochaTestMongoDB(上)

這邊再確定id是驗證過valid的
並且有從DB中(findById(id))取到此筆id資料則回傳

(3) .expect((res) => {expect(res.body.todo.text).toBe(todos[0].text);})
這邊比較資料內容是否正確
並結束

should return 404 if todo not found
此為驗證id正確
但是在DB中找不到資料的情況

(1).get(/todos/${hexId})
在var hexId = new ObjectID().toHexString();建立好驗證後的id
將其輸入到網址/todos/${hexId}

(2) .expect(404)
轉換到server.js
進行id的驗證正確與否判斷

儘管會通過
卻會因為資料無法在DB中取得回傳status 404

判斷是404後並end()結束測試

should return 404 for non-object ids

(1) .get(‘/todos/123abc’)
此情況為在id驗證時就不通過的情況
我們藉由url params去取得id
所以在這邊我們只要輸入任意的非驗證過的url就能進行測試
這邊輸入123abc

(2) .expect(404)
在server.js的if (!ObjectID.isValid(id))驗證失敗
回傳404
並end()結束

以上介紹為/GET上的所有測試

DELETE測試

DELETE /todos
進行三個測試案例

  • should remove a todo
  • should return 404 if todo not found
  • should return 404 if object id is invalid

有發現嗎？
基本上和GET /todos/:id測試很像

should remove a todo

基本上和should return todo doc做的事一模一樣
方法改用delete()

進入server.js的app.delete('/todos/:id')

刪除成功回傳200
並且比較id是否為同筆

比較特別的是
expect提供toNotExist()方法
檢驗DB中此筆資料是否已刪除

should return 404 if todo not found
should return 404 if object id is invalid

此兩個測試案例都和GET時一模一樣
並且沒有資料刪除
所以DB也不用進行測試

PATCH測試

進行兩個測試案例

  • should update the todo
  • should clear completedAt when todo is not completed

以測試completedAt有無作為分別

should update the todo

todos[0]與todos[1]分別有completedAt與沒有completedAt
用patch()進行更新並且send({})要更新的資料
轉換到server.js

成功的話status 200
接著測試更新後從DB回完的資料
分別測試內容text、completed是否為true、completedAt是否有並且是數字num
最後end()

should clear completedAt when todo is not completed

差別在於send({})給予completed: false
一樣會進行更新
測試部分一樣是內容、completed與completedAt
並且end()

小結

以上我們完成了所有express的CRUD測試
這些都是最基本卻也最重要的測試
思考需要進行哪些測試時
筆者認為
某個函式的目的為何
如此重要的事情一定要進行測試

而測試又牽扯到各種面向
以本例子來說
牽扯到前端、後端、DB
單元測試的語法就必須同時考量到這些面向的語法
光是找尋有哪些語法能用的過程就相當繁瑣
更不用說後續寫成測試語言
不熟悉單元測試的初學者絕對是要多付出更多的時間完成
這是單元測試不容易入門的地方
也是筆者需要再多加努力的地方
